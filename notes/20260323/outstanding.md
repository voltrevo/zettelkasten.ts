# Outstanding Tasks & Ideas

## Immediate

### Docker token bootstrap
Agent container needs the server's devToken but tokens are generated by the
server on first run. Current workaround: manually read server config and set
ZTS_DEV_TOKEN. Options: shared volume, server bootstrap endpoint, init
container that generates and distributes tokens.
Files: docker/entrypoint-agent.sh, docker/docker-compose.yml

## Near-term

### Summary upgrade ({{summary}} template)
Agents follow the previous iteration's "next useful atom" suggestion as an
instruction rather than context, even when assigned a different goal. Replace
{{summary}} with a digest of recent activity — last 3-5 summaries, recent
published atoms, goal progress. Gives broader context without anchoring on
one agent's plan.
Files: src/worker.ts (template expansion), src/worker-prompt.md

### Agent container watchdog
Agents can run arbitrary `deno run`/`deno eval` that spin forever (the
inflateDynamic infinite loop incident). The checker timeout only covers test
subprocess runs. Need a container-level mechanism to kill long-running or
high-memory deno/node processes. Could be a simple cron loop, cgroup limits,
or a sidecar monitor.
Files: docker/Dockerfile.agent, new file for watchdog

### Main atoms and exec
Document in the agent prompt how `main` atoms work — atoms that export a
`main(cap)` function callable via `zts exec <hash>`. Explain the connection
between main atoms, the Cap convention, and `zts exec`. Currently shelved.
Files: src/worker-prompt.md

### Goal completion guidance
The prompt mentions `zts goal done` in conventions but could better explain
when and how to mark a goal complete. Should the agent assess completeness
against the goal description? What constitutes "done" for open-ended goals?
Files: src/worker-prompt.md

## Future

### AST scan for banned platform APIs
Enforce at validation time (not just via prompt instructions) that atoms don't
use banned globals like crypto.subtle, CompressionStream, fetch, WebSocket.
Walk the AST and reject atoms that reference them.
Design notes: notes/20260322/global-ban-enforcement.md
Files: src/validate.ts, src/server.ts

### UI template documentation
Show available template variables ({{cli-help}}, {{goal}}, {{summary}},
{{server-url}}) with descriptions in the prompt editor UI. Currently the
operator has to know them by reading the code.
Files: ui/src/pages/prompts.ts

### Cost-effective agent model selection
Determine the most cost-effective model that can build atoms properly and
autonomously. Haiku is cheap but makes more mistakes (btype bug, fixedTables
errors). Opus is better but expensive. Sonnet may be the sweet spot. Need
systematic comparison.

### Docker image rebuild workflow
Currently patching files into running containers. Should have a smooth
rebuild-and-restart flow that preserves data volumes. The compose file
supports this but the operational workflow isn't documented.
Files: docker/README.md

### Worker workspacesDir validation
workspacesDir should be validated as an absolute path in worker.ts to avoid
CWD-dependent behavior.
Files: src/worker.ts
